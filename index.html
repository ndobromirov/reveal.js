### [The 12 factored application](https://12factor.net) - Set of rules between OS and app. - Multiple pros and cons.
### I. Codebase - Managed in version control - 1 code-base is 1 application with many deploys ![scheme](images/codebase-deploys.png)
### II. Dependencies - Always explicit - Isolated - Declarative - PHP - composer
### III. Configuration - Strict separation between code and config. - Comes from environment - always! - Scalable (in deploys count)
### IV. Backing services - Any service the app consumes over the network - No distinction between local or remote ![scheme](images/attached-resources.png)
### V. Build, release, run! - Strong separation between phases. - Ordered by time or incremental. - Immutable releases. - Atomic and fast code reverts. ![scheme](images/release.png)
### VI. Processes - Stateless and Share-nothing - Data persistence to a backing service - Avoid sticky sessions.
### VII. Port binding - The twelve-factor app is completely self-contained. - It exposes a service through port binding (HTTP). - Allows for the app to become a backing service.
### VIII. Concurrency - Based on the unix process model. - Process types based on work type. - Utilize OS native process manager. ![scheme](images/process-types.png)
### IX. Disposability - Appâ€™s processes are disposable! - Minimize startup time - Shut down gracefully on SIGTERM - Sudden death - transactions, etc.
### X. Dev & prod parity - Avoid gaps in: time, people, tools - Use same backing services! - Avoid adapters for backing services.
### XI. Logs - Log streams should be streamed to ```stdout``` - Avoid app level log storage or routing code.
### XII. Admin processes - One-off processes for maintenance. - Same env as the log-running processes. - Example: migrations, updates.
